## Approach

### TD-08 — mechanical swap onto `getAuthenticatedUser()`

For each of the 6 routes, replace:

```ts
const session = await getServerSession(authOptions);
if (!session?.user?.id) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

with:

```ts
const auth = await getAuthenticatedUser();
if (auth.error) return auth.error;
```

and drop the now-unused `getServerSession` / `authOptions` imports, adding
`import { getAuthenticatedUser } from "@/lib/utils/auth";`.

Four constraints the Coding agent must hold:

1. **Do not bind `auth.userId` to a variable in any of the 6.** None of them
   uses the user id — every one of them is a pure "is anyone signed in" gate on
   a symbol-keyed public-market read. Adding `const { userId } = auth` would be
   an unused binding and a new `lint` warning. (This is why the TECH_DEBT
   recommendation's "and read `auth.userId`" clause does not apply here — no
   route has a reader.)
2. **Guard position within the handler is unchanged.** In all 6 the guard is the
   first statement inside the existing `try`. It stays there. In particular, in
   the 4 rate-limited routes (`market/search`, `news/[symbol]`,
   `research/[symbol]/intrinsic-value`, `sentiment/[symbol]/history`) the guard
   runs **before** `checkRateLimit` today — preserve that order exactly. Moving
   the guard after the limiter would change which of two error responses an
   unauthenticated flooder receives, i.e. an observable behavior change smuggled
   into a mechanical refactor.
3. **`NextResponse` imports must stay** in all 6 — every route still uses
   `NextResponse.json` for its success and/or 500 path. Only
   `getServerSession`/`authOptions` become unused.
4. **`market/search`'s outer `catch` is a mock-data fallback, not an error
   path** (`app/api/market/search/route.ts:59-79` returns HTTP 200 with mock
   stocks). The guard's returned 401 is a `return`, not a `throw`, so it exits
   before that catch and is unaffected — but the Coding agent must not
   "improve" this catch while in the file. It is out of scope (guardrail 2: no
   while-I'm-here edits); if it looks wrong, log it in the summary.

**Why swap rather than leave alone:** the duplication is the entire debt. With
one definition, a future change to the auth check (e.g. adding a role claim, or
changing the 401 body) lands everywhere at once instead of in 16 of 22 places.
The swap also lets `AGENT.md`'s convention line stop carrying a "7 older routes
still inline it" carve-out (which is itself stale — it says 7, the real count
is 6).

### TD-15 — migrate the quote fetch to `useQuery`, and split error from pending

#### The template

Two templates exist and they agree. In-file, `positionQ`
(`page.tsx:91-101`) and `transactionsQ` (`:108-117`) establish the shape:
`queryKey: [name, symbol]`, an inline async `queryFn` that `fetch`es and throws
`new Error("Failed to fetch X")` on `!res.ok`, `enabled: !!symbol`, and an
explicit `staleTime`.

More directly, the sibling page `app/(dashboard)/portfolio/[ticker]/page.tsx:63-70`
**already runs the identical query against the identical endpoint**:

```ts
const { data: quote, isLoading: quoteLoading } = useQuery({
  queryKey: ["quote", ticker],
  queryFn: async () => {
    const res = await fetch(`/api/market/quote/${ticker}`);
    if (!res.ok) throw new Error("Failed to fetch quote");
    return res.json();
  },
  enabled: mounted && !!ticker,
  refetchInterval: 30000,
});
```

**Adopt `queryKey: ["quote", symbol]` — deliberately the same key as the sibling
page.** The two pages render the same company header and the same 4-col quote
stat card off the same endpoint (DESIGN.md, "Position detail" — "now
header-for-header identical to Research detail"). Sharing the cache entry is the
correct outcome: navigating `/research/AAPL` → `/portfolio/AAPL` paints the
header instantly from cache instead of refetching. This is exactly the
"shared invalidation" TD-15 names as the missing property.

**Do not copy `refetchInterval: 30000` onto the research page's query.** Reason:
`refetchInterval` is a per-observer option in React Query v5, not a
cache-entry-wide one, so declining it here does not disable polling on the
position-detail page when that page is mounted, and it does not add polling to
the research page. The research detail page is a read-and-analyze surface, not a
live-position monitor; the position page polls because it drives live P/L.
Keeping the difference is intentional — record it as an inline comment so a
future reader does not "fix" the asymmetry.

Specify:

```ts
const quoteQ = useQuery({
  queryKey: ["quote", symbol],
  queryFn: async () => {
    const res = await fetch(`/api/market/quote/${symbol}`);
    if (!res.ok) throw new Error("Failed to fetch quote");
    return res.json();
  },
  enabled: !!symbol,
  staleTime: 5 * 60 * 1000,
});
const quote = quoteQ.data;
```

`staleTime: 5 * 60 * 1000` matches both in-file siblings (`:100`, `:116`) rather
than the `QueryClient` default of 60s (`components/providers.tsx:11`) — match
the file, per the task brief. Keeping a local `const quote = quoteQ.data` alias
means every one of the ~14 existing `quote?.`/`quote.` reads in the JSX is
untouched, which keeps this diff honestly small and reviewable.

#### What depends on the two `useState`s — audited

Checked every reference before specifying removal:

- **`loading`** — referenced in exactly one place, the early return at
  `page.tsx:130-136` (full-page centered `Loader2` spinner). Nothing else reads
  it. Replaced by `quoteQ.isPending` (see below).
- **`setLoading`** — only inside `fetchStockData` (`:74`, `:84`).
- **`quote`** — read in the header (`:147`, `:151`), the whole 4-col stat card
  (`:176-209`, gated on `quote &&`), and passed into five tab components:
  `Overview` (`:231-239`, gated on `quote &&`, reads `.name`/`.price`/`.currency`),
  `TechnicalAnalysis` (`:242`, `quote?.currency`), `FundamentalAnalysis` (`:246`,
  `quote?.currency`), `AnalystRatings` (`:250`, `quote?.price`/`quote?.currency`),
  `IntrinsicValue` (`:253-255`, gated on `quote &&`), `TransactionsTab` (`:258`,
  `quote?.currency`), `NewsFeed` (`:261`, `quote?.name`).
- **`setQuote`** — only inside `fetchStockData` (`:79`).
- **No derived value, `useMemo`, `useEffect` dependency, or conditional other
  than the ones above depends on either state.** `effectiveTab` (`:128`),
  `overviewContext` (`:102`), and `showPositionsTab` (`:118`) are all derived
  from `positionQ`/`transactionsQ`/`activeTab` — untouched by this change.

So: delete `useState<any>(null)` for `quote`, delete `useState(true)` for
`loading`, delete the `useEffect` (`:66-71`) **and its
`// eslint-disable-next-line react-hooks/exhaustive-deps` comment**, delete
`fetchStockData` (`:73-86`), and drop `useEffect`/`useState` from the React
import **only if `activeTab` is the sole remaining `useState`** — it is not:
`activeTab` (`:64`) still needs `useState`, so the import becomes
`import { useState } from "react";` (drop `useEffect` only). Getting this
wrong is a typecheck failure, so the Verify block catches it.

#### Error vs. pending — the visual decision, and why no Designer stage is needed

Today the two states are conflated: `loading` starts `true`, the fetch's catch
only logs, `finally` sets `loading` false, so a failed fetch renders exactly
what a successful-but-empty fetch renders — the header with the bare `symbol`
as the H1, no stat card, and tabs whose components receive `undefined` props.

Specify three distinct states, in this order, before the main return:

1. **Pending** — `if (quoteQ.isPending) { …existing centered Loader2… }`.
   Markup unchanged from `:130-136`, only the condition changes. Note this is
   `isPending`, not `isLoading`: they differ only when the query is disabled
   with no data (`enabled: !!symbol`), and on this route `symbol` comes from
   `useParams()` and is always present, so they are equivalent here —
   `isPending` is specified because it is the property TD-15's recommended fix
   names and it reads correctly if `enabled` is ever tightened.

2. **Error** — new. `if (quoteQ.isError) { … }`. Render:

   ```tsx
   <div className="flex h-32 items-center justify-center rounded-lg border border-border bg-card text-mut">
     <AlertCircle className="mr-2 h-4 w-4" />
     Unable to load quote data
   </div>
   ```

   **This uses an existing DESIGN.md-governed pattern, not a new one.** It is
   the app's established full-card failure state, used verbatim (same classes,
   same `AlertCircle` + `mr-2 h-4 w-4`, same `text-mut` copy voice) at
   `components/technical-analysis.tsx:53-59`, `components/fundamental-analysis.tsx:91-98`,
   and `components/intrinsic-value.tsx:65-72`. Every token in it is already in
   DESIGN.md's table (`--border`, `--card`, `--mut`); DESIGN.md line 255 /
   line 268 already govern `--mut` as the null/unavailable treatment, and its
   "Edge cases carried over unchanged" note (line 2012) treats
   empty/loading/error states as existing behaviors whose presentation the
   Meridian pass already settled. The copy follows the in-app convention
   "Unable to load <thing>" (`portfolio-chart.tsx:137` "Unable to load
   performance data", `fundamental-analysis.tsx:95`, `technical-analysis.tsx:57`).
   **No new token, no new component, no new copy idiom → no Designer stage.**

   One deliberate departure to record: the error card is rendered **in place of
   the whole page body**, wrapped by the existing back-link and company header
   so the user is not stranded on a blank screen with no way back. Concretely:
   render the `← The research desk` `Link` (`:140-142`) and the header block
   (`:144-174`) with `{symbol}` as the H1 (the existing `quote?.name || symbol`
   fallback already produces this), then the error card in place of the stat
   card + tab bar + tab body. The sibling page's "Position not found" state
   (`portfolio/[ticker]/page.tsx:163-176`) sets the precedent that a page-level
   dead-end keeps a navigation affordance.

3. **Resolved** — the existing return, unchanged except that `quote` now comes
   from `quoteQ.data`.

`AlertCircle` must be added to the existing `lucide-react` import (`:8`,
currently `Loader2, Plus, Star`).

**Deliberately NOT in scope:** a retry button. No page in this app offers one
today (the three tab-level failure cards do not), React Query's
`refetchOnWindowFocus` is off globally (`providers.tsx:12`) so a manual reload
is the current recovery idiom everywhere, and adding a first-of-its-kind
affordance *is* a Designer decision. If the owner wants one, that is a separate
pass. Logged in `## Assumptions`.

#### Secondary consequence worth stating

Migrating to React Query means a failed quote now propagates as a thrown error
inside `queryFn` → `isError`, so the previously-silent `!res.ok` case (401 after
session expiry, 500 from the route's own catch) becomes visible for the first
time. That is the point of the fix, but it is a **user-visible behavior change**
on a path that previously rendered a degraded-but-quiet page: this is the one
part of this plan that is not behavior-preserving, and it is intentional.

### Testing note for TD-15 — why the error test is route-level

CLAUDE.md requires a happy-path and a meaningful-failure test per modified
function. For TD-15 the modified surface is a React component, and **this repo
has no component-test seam**, verified during planning:
`vitest.config.ts` sets `environment: "node"` and
`include: ["**/*.test.ts"]` (not `.tsx`); there is no `jsdom` or `happy-dom`;
`@testing-library/react` is not installed (`node_modules/@testing-library` does
not exist); `@vitejs/plugin-react` is in `package.json` but is **not wired into
`vitest.config.ts`**; and there are zero `*.test.tsx` files in the repo. ADR-22
records this same constraint and its precedent resolution — the settings page's
gate logic was extracted to a pure module (`lib/utils/scoring-weights-settings-gate.ts`)
precisely because "no component-test seam exists."

Two options were considered and one rejected:

- **Rejected — install jsdom + `@testing-library/react` and write a component
  test for the three render states.** This is the technically complete answer,
  but standing up a component-testing stack is a test-infrastructure decision
  with repo-wide consequences (a second vitest environment or a projects split,
  a new render-test convention every future UI plan inherits, ~3 new
  dependencies) attached to a Low-severity Low-effort debt item. That is an
  architectural call, not something to slip into a cleanup pass. Logged as a
  new `TECH_DEBT.md` item (TD-38, below) so the gap is tracked rather than
  silently accepted.
- **Chosen — test the failure *condition* at the route level, and verify the
  render states manually (M1/M2).** The genuinely new logic in TD-15 is "a
  non-ok quote response must become a visible error." Task 6 asserts the route
  actually produces a non-ok response when its service fails (making the new
  branch reachable rather than theoretical); the `queryFn`'s `!res.ok → throw`
  is three lines copied verbatim from two existing call sites; and the render
  branch itself is asserted by M2. This is the same shape as ADR-22's fallback.

## Assumptions

- **A1 — Sharing `queryKey: ["quote", symbol]` with `portfolio/[ticker]/page.tsx`
  is desirable, not a collision to avoid.** Both consume the same endpoint for
  the same rendering purpose, and DESIGN.md states the two headers are
  header-for-header identical. If the owner would rather the two pages hold
  independent cache entries, the research page's key becomes
  `["quote", symbol, "research"]` — a one-line change, and the only consequence
  is losing the cross-page cache hit in M1.
- **A2 — No retry affordance on the error state.** No page in this app has one
  today; adding a first-of-its-kind control is a Designer decision, deliberately
  out of scope here. Manual reload remains the recovery path.
- **A3 — `staleTime: 5 * 60 * 1000` for the quote query**, matching the two
  in-file siblings rather than the 60s `QueryClient` default. A quote can
  therefore be up to 5 minutes stale on the research page. This is a research/
  analysis surface, not a trading surface — the position-detail page keeps its
  30s polling for the live-P/L case. If the owner wants fresher quotes on
  research, dropping `staleTime` to inherit the 60s default is a one-line
  change.
- **A4 — TD-15 makes previously-silent quote failures visible**, which is a
  real user-visible behavior change (a page that used to render degraded-but-
  quiet will now render an error card). This is the intended fix, not a
  regression, and approving this plan approves that change.
- **A5 — The 2 new auth tests (Task 3) and the 1 new route test (Task 6) are
  the full new test scope.** No backfill of the other 4 routes' happy paths is
  attempted — 4 of the 6 already have 401 coverage, and broad API test backfill
  is TD-07's scope, not this pass's.
- **A6 — No component-test stack is installed in this pass** (see the Testing
  note); the gap is logged as TD-38 instead.

## Open decisions

None. Both items resolve against verified code and existing DESIGN.md-governed
patterns; no new token, component, or copy idiom is introduced, so no Designer
stage is required before coding.
