## Findings

### PT-I1 — ISSUE
**File:** `components/research/transactions-tab.tsx:92-116`; `app/(dashboard)/portfolio/[ticker]/page.tsx` (removed header block, diff lines ~194-278); `DECISIONS.md` ADR-18 (Decision + Evidence)
**Problem:** Realized P/L is no longer displayed anywhere in the UI for a position that has sold shares. The old `/portfolio/[ticker]` header rendered a conditional "Realized P/L" cell (the removed `hasRealizedPL` block: `const hasRealizedPL = position.realizedPL !== undefined && position.realizedPL !== 0`). Both the plan (Task 2 / the header-relocation description) and ADR-18's Decision text explicitly list "Realized P/L" among the position-specific stats that "move out of the header into a tab." But the shared `TransactionsTab` stat band renders only **Shares held / Average cost / Market value / Unrealised P/L** — there is no Realized P/L cell, and `PositionResponse` in `transactions-tab.tsx:16-23` does not even carry a `realizedPL` field. For a fully-sold (`quantity === 0`) position the band is suppressed entirely, so realized gains are doubly hidden. Net effect: a user who partially or fully sold a position can no longer see their realized P/L in the app. This is a real (if low-severity) capability regression, and it makes ADR-18's Evidence/Decision claim ("Realized P/L … move into a tab") inaccurate — the number moves nowhere; it disappears.
**Recommendation:** Either (a) add a Realized P/L cell to the `TransactionsTab` "held" stat band (and decide whether to show it in the "closed" state, where it is arguably most relevant — a closed position's whole value is its realized result), extending `PositionResponse` to include `realizedPL` from the already-returned position payload; or (b) if dropping the UI surface for Realized P/L is an accepted scope reduction, correct ADR-18's Decision/Evidence text to say Realized P/L is *removed* rather than *relocated*, and note the gap in `TECH_DEBT.md`. Do not leave the ADR claiming a relocation that did not happen. This is a UI/display change only — it does not touch the realized-P/L computation or the sell/FIFO flow.

### PT-S1 — SUGGESTION
**File:** `app/(dashboard)/research/[symbol]/page.tsx:124-129`; `app/(dashboard)/portfolio/[ticker]/page.tsx:120-125`
**Problem:** The `activeTab` fallback is a post-render `useEffect` keyed on `[showPositionsTab]`. Because the effect fires after the render commit, there is a one-render window in which `activeTab === "transactions"` while the "transactions" entry has already been filtered out of `tabs`. In that window the tab body still renders (`{activeTab === "transactions" && <TransactionsTab …/>}`) even though no active tab button exists. In practice this is essentially unreachable: `activeTab` starts at `"overview"` on both routes, the tab only disappears when transactions go from present→empty (only via Delete on the portfolio route, which immediately `router.push("/dashboard")`), so the flagged transient never actually paints for a user. Correct as written, but the guard is defensive-only and relies on that timing.
**Recommendation:** Optional hardening — derive the effective active tab during render instead of correcting it after commit, e.g. `const effectiveTab = tabs.some(t => t.value === activeTab) ? activeTab : "overview"` and switch the body/`className` on `effectiveTab`. This removes the theoretical mismatch window entirely and drops the need for the `eslint-disable` on the effect. No behavioural change for real users.
