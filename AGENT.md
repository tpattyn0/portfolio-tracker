# AGENT.md — Meridian
> Project-specific conventions and constraints. Generic guardrails are in `CLAUDE.md` — not here.

## Conventions

- **Route handlers stay thin, delegate to `lib/services/*.service.ts`** (ADR-3). Don't put business logic directly in `app/api/**/route.ts`.
- **Auth guard pattern for new routes:** use `getAuthenticatedUser()` / `getAuthenticatedUserWithPortfolio()` from `lib/utils/auth.ts` — as of 2026-07-23 (TD-08, ADR-28) it is the **only** auth-guard pattern under `app/api/**`; the last 6 inline `getServerSession(authOptions)` + manual-401 routes were swapped onto it. A `grep -rn "getServerSession" app/api --include=route.ts` returning any hit in a `route.ts` file is a regression — do not reintroduce the inline variant.
- **Ownership scoping is done correctly and consistently** — `getAuthenticatedUserWithPortfolio()` derives `portfolioId` from the session, position lookups are scoped by the compound `portfolioId_ticker` key, and wishlist mutations gate on a `findFirst` scoped to the user's own wishlist before writing (`app/api/portfolio/positions/[ticker]/sell/route.ts:28-39`, `lib/services/wishlist.service.ts:110-156`). Follow this pattern for any new mutation route; a reviewer should treat a route that skips it as a deviation worth flagging.
- **Rate limiting:** use `checkRateLimit(request, scope, maxAttempts, windowMs)` from `lib/middleware/rate-limit.ts` for unauthenticated or third-party-fanout routes. It's in-memory and per-instance (see fragile surfaces) — fine for now, revisit if deployed multi-instance.
- **Financial calculation services** (`intrinsic-value.service.ts`, `technical-analysis.service.ts`, `fundamental-analysis.service.ts`) are pure/static-method classes reading from cached DB tables or live Yahoo data. Keep them side-effect-free where possible; persistence/caching happens in the route or a thin wrapper, not buried inside the calculation itself.

## Known fragile surfaces

- **`lib/services/sentiment.service.ts`'s constructor throws (via the shared `createGeminiClient` factory) if `GEMINI_API_KEY` is unset, and a singleton instance is created at module scope.** Importing this module anywhere without the env var set crashes at import time, not call time. Be careful adding new imports of this module in code paths that might run without the key configured (e.g. build-time static analysis, scripts).
- **`lib/services/wishlist.service.ts` previously made self-referential HTTP calls** to its own `/api/news/[symbol]` and `/api/research/[symbol]/intrinsic-value` routes. Fixed 2026-07-17 (AUD-05/TD-10): both now call `newsService.getAnalyzedNewsForSymbol()` and `IntrinsicValueService.calculateIntrinsicValue()` directly. The self-fetch had silently 401'd since those routes gained required auth (ONB-05, 2026-07-16) — `sentimentScore`/`intrinsicScore` were always `null` and the composite score always substituted a neutral 5 for 40% of its weight. If wishlist composite scores seem wrong, verify this path is still calling the services directly (not reverted to `fetch`).
- **`lib/services/fundamental-analysis.service.ts:84`** — cache freshness is gated by an exported `SCORING_VERSION` integer constant (TD-11, `plans/2026-07-23-lib-cleanup-batch.md`, ADR-27), replacing the old hardcoded `latestMigrationDate` date literal. Persisted per-row inside the existing `scoreDetails` JSON column (no schema migration — no `FundamentalData` column exists for it). A cached row is fresh only when `scoreDetails.scoringVersion` is greater than or equal to the current `SCORING_VERSION` (`>=`, so a row written by a newer deployment is not clobbered on rollback) **and** `lastUpdated` is within 24h — the two gates are independent, both must pass. Any future change to extraction/scoring logic that would make previously-cached rows wrong needs `SCORING_VERSION` bumped — do not revert to a date literal, and do not persist the version as a top-level DB column without a Planner-scoped migration.
- **Gemini client construction is centralized in `lib/services/gemini.ts`'s `createGeminiClient`/`getGeminiApiKey`** (TD-12, `plans/2026-07-23-lib-cleanup-batch.md`, ADR-27) — both `sentiment.service.ts` and `app/api/insights/portfolio/route.ts` now call the shared factory instead of each constructing their own `GoogleGenerativeAI` client. `new GoogleGenerativeAI` must not reappear outside `lib/services/gemini.ts` — a `grep -rn "new GoogleGenerativeAI" app lib` should find exactly one hit, the factory itself. The two call sites' **missing-key behavior deliberately still differs** and this is by design, not an oversight: `sentiment.service.ts`'s constructor throws (`createGeminiClient()` with no args, defaulting to `getGeminiApiKey()`), while the insights route reads the key via `getGeminiApiKey()` first and returns its own graceful 200 placeholder without ever calling `createGeminiClient` when the key is absent (AUD-10 — that branch must stay untouched). Both call sites also import the shared `GEMINI_MODEL` constant (unchanged since `plans/2026-07-20-gemini-model-update.md`) — if Gemini retires a model again, change `GEMINI_MODEL` in `lib/services/gemini.ts`, not the two call sites.
- **`IndustryComparison` table is read but never written** — `intrinsic-value.service.ts` reads it for peer-relative valuation but nothing populates it, so those methods fall back to hardcoded defaults (`industryPE = 15`, `industryPB = 1.5`). If this table is ever seeded, re-verify the fallback logic still behaves correctly.
- **`app/api/portfolio/closed-positions/route.ts`** — recomputes on every request (no caching). As of 2026-07-17 the FIFO matching and realized-P/L math live in `lib/services/realized-pl.service.ts`, shared with the sell route (fixed AUD-03: the two surfaces previously disagreed — avg-cost vs FIFO, inconsistent fee treatment; and AUD-04: `isPartial` was dead code because the loop `continue`d past unmatched sells instead of computing it). Both routes now have unit tests for the shared helper. Treat changes here carefully regardless; a bug is a silent wrong number shown to the user (win rate, realized P/L).
- **`app/api/portfolio/positions/[ticker]/sell/route.ts`** — matching a sell against BUY lots is not enough on its own (AUD-FIX-01, 2026-07-17): the route must also replay every **prior SELL** on the position through `matchFifoLots` first to deplete the lots before matching the new sell, or a second sell re-matches against already-consumed lots and `portfolio.realizedPL` silently disagrees with the closed-positions total. The closed-positions route gets this for free because it holds one `buyLots` array and depletes it in a single pass across all sells in chronological order — the sell route re-derives lot state from scratch on every call, so it must explicitly reconstruct "state as of just before this sell" by replaying prior sells. If you ever change how lots are loaded here, keep the prior-sell replay; `resolveCostBasisWithFallback` in `realized-pl.service.ts` is the shared unmatched-remainder fallback used by both routes — change the fallback logic there, not per-route.
- **`lib/services/position.service.ts`** (added 2026-07-17, AUD-02) — single source of truth for "buy more of an existing position": Decimal arithmetic, wrapped in the caller's `$transaction`, recalculates portfolio totals. Used by both `POST /api/portfolio/positions` (existing-position branch) and `POST /api/portfolio/positions/[ticker]/buy`. If either route's buy behavior needs to change, change it here — do not reintroduce a second implementation.
- **`lib/services/realized-pl.service.ts`** (added 2026-07-17, AUD-03; extended AUD-FIX-01/03; extended again 2026-07-19, ADR-18/PT-I1) — single source of truth for FIFO lot matching, the unmatched-remainder average-cost fallback (`resolveCostBasisWithFallback`), and realized P/L (including buy and sell fees). Used by the sell route, the closed-positions route, and now `GET /api/portfolio/positions/[ticker]` via `computePositionRealizedPL(transactions, avgCostBasis)` (sums realized P/L across one position's SELL transactions). `matchFifoLots` mutates its `lots` array in place — pass a fresh array per call, and remember that "call it once per prior sell, then once for the new sell" is how the sell route reconstructs cumulative depletion (see the sell-route fragile-surface entry above); `computePositionRealizedPL` builds its own fresh `buyLots` array per call so it's unaffected by this, but if you add a caller that reuses lots across multiple `computePositionRealizedPL` calls, remember the mutation.
- **`ComponentErrorBoundary` coverage is inconsistent** — only wraps sections on `dashboard/page.tsx` and `portfolio/[ticker]/page.tsx`. `research/[symbol]/page.tsx`, `research/page.tsx`, `wishlist/page.tsx`, `portfolio/add/page.tsx`, and `portfolio/closed-positions/page.tsx` have no boundary around components that depend on external API calls. A runtime error there crashes the whole page. `components/error-boundary.tsx` also exports a full-page `ErrorBoundary` class that is never used anywhere, including `app/layout.tsx`.
- **`research/[symbol]/page.tsx` is now fully React-Query-driven (2026-07-23, TD-15, ADR-28)** — its quote fetch (`quoteQ`) moved from a manual `useEffect`/`fetchStockData`/`useState` trio to `useQuery`, **deliberately sharing `queryKey: ["quote", symbol]`** with the identical query on `portfolio/[ticker]/page.tsx:64-72` (both pages render the same header/stat card off the same endpoint; sharing the cache entry is intended, not a collision) — but **deliberately NOT sharing that sibling's `refetchInterval: 30000`** (a per-observer React Query v5 option; this page adds no polling observer of its own, but because the `["quote", symbol]` entry is shared, a concurrently-mounted sibling's 30s poll refreshes the shared cache entry and this page — as a subscriber to that entry — will observe the update; harmless on this read-only surface, but not the "never updates" guarantee the wording could be misread as). If a future edit makes the two queries diverge in shape, keep the key shared unless the owner decides the pages should hold independent cache entries. **The page's `isPending`/`isError`/resolved states must stay distinct** — a future edit must not re-collapse the new `isError` branch (back-link + header + the shared "Unable to load quote data" failure card) back into the pending/resolved cases, or it silently regresses TD-15's fix (a failed quote fetch rendering identically to a successful-but-empty one). It still (2026-07-18, MRD-Q1) uses a second, narrower `useQuery` for the `GET /api/portfolio/positions/[ticker]` ownership lookup `components/research/transactions-tab.tsx` already makes (404 = not held), driving the Overview tab's `context` prop (`overviewContext`) — this call site and `transactions-tab.tsx` still duplicate the same fetch/404 logic inline rather than sharing a hook; if a third consumer needs "is this symbol owned", extract a shared `usePositionQuery(symbol)` hook instead of a third copy.
- **Rotating any key in `.env`/`.env.local` can break `npm run verify`** — a new secret value may match a *different* gitleaks rule than the old one (the rotated Gemini key moved from `gcp-api-key` to `generic-api-key`, twice now: `.env` on 2026-07-17, `.env.local` the same day). The fix is a new narrow fingerprint in `.gitleaks-local/.gitleaksignore` for **each** env file carrying the key — never a path allowlist. Read that file's header before touching it.
- **`.next/` was tracked in git until 2026-07-16** — if you ever see hundreds of unrelated `.next/*` diffs in `git status`, someone re-added it; re-run `git rm -r --cached .next`. Do not `git add -A` in this repo.
- **The color token system (`app/globals.css` + `tailwind.config.js`) is load-bearing for all 20 `components/ui/*` files (ADR-8, 2026-07-17 Meridian overhaul).** Every shadcn CSS variable (`--background`, `--card`, `--border`, …) now holds a complete color value (hex or `oklch(...)`), **not** an HSL triple — `tailwind.config.js` maps them as `'var(--x)'`, with the `hsl()` wrapper removed. Do **not** re-add `hsl(var(--x))` to `tailwind.config.js`: it cannot parse hex or oklch, and re-adding it would silently break every component that consumes `bg-card`/`border-border`/etc. (all 20 files in `components/ui/`, Radix primitives included). If `npx shadcn add` ever generates a new component, its own inline `hsl(var(--x))` usage (if any) needs adjusting to match this convention — check the generated file against `components/ui/button.tsx` for the expected pattern (Tailwind alias, no inline `hsl()`). The Meridian-only tokens (`--mut`, `--line2`, `--fill`, `--up`, `--dn`, `--amber`, `--btnbg`, `--btnfg`) are also exposed as Tailwind color aliases (`text-mut`, `bg-fill`, etc.) — reference DESIGN.md's token table before adding a new one-off value.
- **The dashboard performance chart (`components/portfolio-chart.tsx`) is a hand-rolled inline SVG, not Recharts** (ADR-10) — it calls `buildPath`/`buildAreaPath` from `lib/utils/chart-path.ts`, a pure Catmull-Rom→bezier path builder with its own test file. If you change the chart's visual behavior, change the math in `chart-path.ts` (and its tests) — do not inline path-building logic back into the component. **`buildPath`'s Catmull-Rom control points can overshoot past a series' own min/max vertex** (`plans/2026-07-20-perf-graph-dip-clipping-fix.md`) — with a dip immediately followed by a steep recovery (a real shape: a portfolio low right before a big new-position-driven rise), the interpolated curve between the dip's vertex and the steep segment can dip *further* than the vertex itself, past the plot floor and off the unclipped (`preserveAspectRatio="none"`) viewBox — this is what "the dip disappears under the x-axis" looked like. Fixed via `buildPath`'s optional `domainMargin` param (both chart components pass `CHART_DOMAIN_MARGIN`, default `0` preserves the old behavior exactly) — see the `gridlineYs()` entry above for the full domain-margining contract. `lib/utils/chart-path.test.ts` has a dedicated `describe` block that samples the actual bezier curve (not just vertices) and asserts the pre-fix shape clips at `domainMargin = 0` and stays inside the plot at `CHART_DOMAIN_MARGIN` — treat a failure there as a real regression, not flakiness.
- **The research-detail Overview/Technical charts (`components/research/detail-price-chart.tsx`) are a second, distinct inline-SVG chart** (ADR-11) — same `buildPath`/`buildAreaPath` primitives as the dashboard hero, but a separate component: no range-morph, and it adds a minimal y-axis (`niceYTicks` from `lib/utils/chart-ticks.ts`) + optional dashed reference lines that the hero chart intentionally does not have. `components/price-chart.tsx` (Recharts) was deleted 2026-07-18 once this component replaced its last importer — Recharts (`recharts` package) is no longer used anywhere in the app. **As of `plans/2026-07-18-meridian-dashboard-detail-fixes.md`, the dashboard hero chart (`components/portfolio-chart.tsx`) now also has hover crosshair+tooltip and y-axis price labels**, ported from this component's container-ref pixel-mapping approach and sharing `niceYTicks` — the two components are no longer distinguished by "has hover/y-axis vs. doesn't"; the hero still has its own 220-tall viewBox vs. this component's 190-tall one (both charts' gridline y-positions are now derived from the plotted series' own domain via `gridlineYs()`, not fixed pixel fractions — see the `gridlineYs()` fragile-surface entry below, `plans/2026-07-20-small-visual-fixes.md`) and no range-morph interruption risk was introduced (the mousemove handler only reads `animatedValues`/`series` state, never touches the RAF ref). Do not merge the two SVG charts or backport the hero's range-morph into this one, or vice versa, without a Designer decision.
- **`components/research/*` shared primitives are the single source of truth for the research-detail tabs' editorial chrome** (ADR-11) — `HeadlineScoreCard`, `ScoreFigure`/`VerdictStamp`, `SubscoreBand`, `GradedMetricRow` are consumed by `overview.tsx`, `technical-analysis.tsx`, `fundamental-analysis.tsx`, `analyst-ratings.tsx`, `intrinsic-value.tsx`, `news-feed.tsx`. If a tab's headline card needs to look different, change the shared component (or add a prop) — do not fork a one-off variant inline in a tab component; that was the exact duplication `HeadlineScoreCard` was extracted to prevent.
- **Score/grade banding is centralized in `lib/utils/score-band.ts`** — `scoreBandClass`/`gradingDotClass`/`metricGrade` are presentational-only (recolor already-computed numbers against fixed thresholds) and must never be changed to alter scoring math; that lives in `lib/services/*-analysis.service.ts`. If a new tab needs a banded figure or grading dot, import these helpers rather than reimplementing the ≥7/4-7/<4 threshold inline.
- **`lib/services/technical-analysis.service.ts` has two entry points now** (`plans/2026-07-18-performance-audit-remediation.md` Task 3) — `calculateIndicators` (the pure, always-fresh calculation) and `getCachedIndicators(symbol, prices, volumes)` (a keyed 60s in-process memo wrapping it, matching `market-data.service`'s history cache TTL). `app/api/market/chart/[symbol]/route.ts` calls the cached wrapper; `wishlist.service.ts` still calls `calculateIndicators` directly (its own historical-data fetch is already 60s-cached in `market-data.service`, so a second cache layer there would key on a different fingerprint per call site for no benefit). If a new caller needs indicators for a stable `symbol`, prefer `getCachedIndicators` — but never change what either method computes; the cache is memoization only, not a second implementation.
- **`app/api/portfolio/route.ts`'s day-change now comes from `getQuote(...).previousClose`, not a per-position `getHistoricalRange` call** (Task 4, same plan) — if a future change needs the *prior official trading-day close* specifically (as distinct from Yahoo's `regularMarketPreviousClose`), that's a deliberate behavior change requiring a fresh parity check, not a silent revert of this fix.
- **`lib/services/wishlist.service.ts`'s five per-item score fetches run concurrently via `Promise.allSettled`, not sequential `await`s** (Task 5) — each dimension's failure is isolated (a rejected promise for one score does not affect the others), matching the pre-parallelization try/catch-per-dimension behavior. Do not reintroduce a sequential chain here; if a new score dimension is added, add it to the same `Promise.allSettled` array.
- **`app/(dashboard)/layout.tsx` must stay a non-blocking, synchronous Server Component — do not reintroduce a server-side session `await` there.** Page-route auth is enforced solely by `middleware.ts` (ADR-2, ADR-16); the layout previously duplicated that check via `await getServerSession(authOptions)` + `redirect`, which blocked the entire dashboard subtree — including the shared `<Navigation>` and every child route's `loading.tsx` Suspense fallback — from painting until the session check resolved. In the App Router, a segment's `loading.tsx` wraps only that segment's children in a Suspense boundary, not the layout above it, so any blocking `await` in this layout's render path re-freezes navigation across the whole group, not just the route being fetched. If a defence-in-depth check is ever wanted here again, it must be non-blocking (e.g. a separate Suspense-wrapped server component reading the session), never a render-gating `await` in the layout itself. See `plans/2026-07-19-meridian-nav-responsiveness.md` and ADR-16.
- **The dev server runs Turbopack (`next dev --turbopack`, ADR-17); `next.config.js` must stay free of a `webpack()` function.** Turbopack does not honor a custom webpack config function the way webpack does — adding one to fix a dev-only problem would silently not apply under `next dev`, while still applying to `next build` (which uses the default bundler, unaffected by this change). If a webpack-only customization is ever genuinely needed, it needs an explicit Turbopack (`turbopack`/`experimental.turbo`) equivalent in the same change, not a bare `webpack()` addition assumed to cover both.
- **`components/research/transactions-tab.tsx`'s live stat band must gate on `position.quantity > 0`, never on `!!position` alone** (ADR-18, `plans/2026-07-19-positions-tab.md`) — a fully-sold-but-not-deleted position still returns a 200 from `GET /api/portfolio/positions/[ticker]` with `quantity: 0` (the sell route updates quantity, it never deletes). Truthy-checking the position record alone would render a misleading zero-value Market value / Unrealised P/L panel for a closed position. The shared, tested helper `getPositionsPanelState(position)` in `lib/utils/positions-tab.ts` is the single source of truth for this three-way distinction (`"held"` / `"closed"` / `"none"`) — if a future caller needs "is this symbol currently held with live stats to show," use this helper, not an inline `!!positionQ.data` check. The companion helper `shouldShowPositionsTab(transactions)` is a separate, coarser gate (any transaction on file, regardless of quantity) driving whether the Positions tab appears at all — do not conflate the two: a symbol can have `shouldShowPositionsTab === true` and `getPositionsPanelState === "closed"` simultaneously (a fully-sold position with transaction history).
- **Realized P/L in the Positions tab is unlike the other stat-band cells: it is NOT gated on `quantity > 0`** (ADR-18 follow-up, owner decision PT-Q1, `reviews/2026-07-19-positions-tab.md` PT-I1) — it is shown in both the `"held"` and `"closed"` states (never `"none"`), gated only on `hasRealizedPL(position.realizedPL)` (hide when exactly `0` or absent, preserving the pre-existing header's `hasRealizedPL` semantic). Market value / Unrealised P/L are meaningless at `quantity === 0` and stay hidden in `"closed"`; Realized P/L is a settled historical figure and is *most* meaningful there — do not extend the `quantity > 0` gate to cover it. `GET /api/portfolio/positions/[ticker]` previously hardcoded `realizedPL: 0` with a stale comment claiming the field wasn't in the schema — it genuinely isn't a `Position` column (only `Portfolio.realizedPL`, a portfolio-wide accumulator, exists), so the route now computes it on read via `computePositionRealizedPL` in `lib/services/realized-pl.service.ts`, reusing the same FIFO algorithm `GET /api/portfolio/closed-positions` already uses per position. If a future change touches either route's realized-P/L math, change it in the shared service — do not reintroduce a third divergent calculation (see the existing `realized-pl.service.ts` fragile-surface entry below for the sell-route/closed-positions history this would repeat).
- **`lib/services/analyst-ratings.service.ts`'s `targetLowPrice`/`targetHighPrice`/`revisions` are now persisted, closing the former cache-hit gap** (ADR-19, `plans/2026-07-20-analyst-revisions-nvda-fix.md`) — `AnalystRating` has `targetLowPrice`/`targetHighPrice` (`Decimal?`) and `revisions` (`Json?`) columns; `saveToDatabase` populates all three on every fresh fetch, and `formatCachedData` (the 24h cache-hit path) reads them back instead of hardcoding `null`/`[]`. **If you change any of the three fields' shape or extraction, keep `saveToDatabase` and `formatCachedData` in sync** — they are the only two places that write/read these columns, and a mismatch (e.g. a new field added to `AnalystRevision` but not round-tripped through JSON) would silently drop data on the next cache hit. `formatCachedData` defensively coalesces (`Array.isArray` check on `revisions`, falsy-check on the two Decimals) so rows written before the migration applied still return `[]`/`null` rather than throwing — do not remove those guards. Revisions are windowed to the last 90 days (inclusive boundary), newest-first, capped to 25 by the pure `filterRecentRevisions(revisions, now, windowDays, cap)` helper applied inside `extractRatings` **before** persistence — the DB always stores the already-filtered set, not the raw Yahoo history (a heavily-covered symbol like NVDA has 982 raw `upgradeDowngradeHistory` entries; only ~32 are within 90 days). **The schema migration (`prisma/migrations/20260719231015_analyst_revisions_low_high`) is additive-only (`ADD COLUMN`, no destructive statements) but was generated with `prisma migrate dev --create-only` and is held for explicit owner `prisma migrate deploy` sign-off** (ADR-6/ADR-14 protocol — dev and prod share one database). Do not apply it without that sign-off, and do not assume it is live in either environment until confirmed.
- **`lib/services/intrinsic-value.service.ts`'s `scenarioLow`/`scenarioHigh` are a method-spread (min/max of the 5 valuation methods' valid values), not a flex-assumptions scenario model** (`plans/2026-07-19-research-tab-fixes.md`, OD-2) — `intrinsicValue` (the confidence-weighted average) is unaffected and remains the Base case; `calculateScenarioRange` requires ≥2 methods with `value > 0` or both return `null` (component renders em-dash). Do not conflate this with a "vary growth/WACC by a band" scenario model — that was the rejected alternative (OD-2 Option B), which would be ADR-worthy if ever built. The "Model assumptions" rows (`components/intrinsic-value.tsx`) show only the three real DCF Lite inputs (`dcfLite.inputs.growthRate`/`terminalPE`/`discountRate`) per OD-1's honest relabel — do not re-add "FCF margin"/"Terminal growth" rows unless the DCF Lite method is genuinely extended to compute them; a permanently-hardcoded em-dash for a non-existent model input was the exact bug this fixed.
- **`components/overview.tsx`'s composite `HeadlineScoreCard` must gate its loading state on all five subscore queries, not just `chartQ`** (`plans/2026-07-20-small-visual-fixes.md`, Issues 2/3) — the composite score and `SubscoreBand` are assembled from five independent `useQuery` calls (chart/technical, fundamentals, analyst, intrinsic, news), each defaulting to a fabricated neutral `5` while its own query is still loading. Gating the card's `isLoading` on only one query (the original bug) let the card paint with a composite built from one real subscore + four fake `5`s, then visibly jump as each remaining query resolved. The fix ORs all five queries' `isLoading` (React Query v5: `isPending && isFetching`, true only during a query's first fetch with no cached data — false for a disabled query with no data, so `intrinsicQ`'s `enabled: currentPrice > 0` doesn't block the gate forever when disabled). If a sixth dimension is ever added to the composite, add its query to this same OR chain — a dimension left out of the gate reintroduces the identical flicker. Do not confuse this with the *resolved*-but-empty case: the neutral-`5` fallback inside each subscore's `useMemo` (keyed on `.isError`, not `.isLoading`) is unchanged, documented behavior and must stay — only the newly-added gate distinguishes "still pending" from "resolved with nothing useful."
- **`lib/utils/chart-path.ts`'s `gridlineYs()` must be passed the same domain (`yMin`/`yMax`/`padding`) used to call `buildPath` for the same series** (`plans/2026-07-20-small-visual-fixes.md`, Issue 4) — both `portfolio-chart.tsx` and `detail-price-chart.tsx` previously drew gridlines/y-axis labels at fixed pixel fractions of the viewBox (`GRIDLINE_Y = [55,110,165]` / `[47,94,141]`) that did not correspond to where `buildPath` actually plots the series' min/max, so a spiky series' extremes could render outside the labelled gridline band. `gridlineYs` closes this by deriving each tick's y-pixel from the identical padded-domain formula `buildPath` itself uses. If either chart's `CHART_PADDING` constant ever diverges from `buildPath`'s default (`8`), pass it explicitly to both — do not let the two drift independently. **As of `plans/2026-07-20-perf-graph-dip-clipping-fix.md`, the domain passed to both `buildPath` and `gridlineYs` is no longer the raw series `[min, max]` — it is `marginDomain(min, max, CHART_DOMAIN_MARGIN)` (both exported from `chart-path.ts`), a small symmetric expansion (`~8%` of range) that gives the Catmull-Rom spline's overshoot (see the next entry) somewhere to go before it clips the plot floor/ceiling.** `niceYTicks(min, max, 3)` — the LABEL values shown to the user — is still called with the **raw, unmargined** `min`/`max` in both components; only the pixel mapping passed to `buildPath`/`gridlineYs` uses the margined domain, so the numbers a user reads never change. **Do not set `CHART_DOMAIN_MARGIN` to `0`** in either chart component (it would silently revert to the old floor-flush behavior and a steep dip-then-spike series would clip below the axis again — the exact bug this fix closed) and do not derive `domainMin`/`domainMax` independently in the two components; both must call the shared `marginDomain()` helper so their domains cannot drift apart, the same single-source-of-truth invariant `gridlineYs` established for the un-margined case. ## Verify

Single command, run before every commit:

```bash
npm run verify
```

This runs, in order: `typecheck` (`tsc --noEmit`), `lint` (`next lint`), `test` (`vitest run`), and `secret-scan` (`gitleaks detect --source . --no-git -c .gitleaks.toml --redact`). All four must pass. `lint` currently has pre-existing warnings (unused imports, a few `any` types) that do not fail the command — do not treat fixing all of them as in-scope for an unrelated change; fix only what you touch.

### Reading CI on a PR

Two checks, and only one of them blocks:

| Check | Blocks merge? | What a red means |
|---|---|---|
| `verify / code-gate` | **Yes — required** | A real failure. Typecheck, lint, tests, or a secret in *this PR's* commits. |
| `verify / secret-history (reporting only — expected red)` | No | **Expected.** It reports the known historical leak (ADR-7 / TD-01). It is red on every run and will stay red until TD-01 is fully resolved. |

So a PR showing one red X and one green is **normal**. Two reds means `code-gate` failed and something is actually wrong. These were previously both named `verify`, which made a real failure indistinguishable from the expected one at a glance — renamed 2026-07-17.

**CI runs `npm run verify:code`, not `npm run verify`** — same typecheck + lint + test gate, minus the local secret scan. CI handles secrets in its own steps instead: `code-gate` scans this PR's commit range, and `secret-history` scans all of history. Neither uses the gitleaks GitHub Action — on a `pull_request` event it only scans the PR's own commits and ignores `fetch-depth`, which is how a green check once coexisted with three live secrets in the first commit. Keep the two in sync: a check added to `verify` belongs in `verify:code` too.

## Git hooks live in `.githooks/`, not `hooks/`

`hooks/` is application source (React hooks — `use-debounce`, `use-price-sync`, `use-toast`), imported as `@/hooks/*` across `app/` and `components/`. The workflow framework's rollout script defaults to putting `pre-commit` and `install.sh` in a root `hooks/` directory; in this repo they were moved to `.githooks/` to keep shell scripts out of a source tree.

Install them on a fresh clone with `bash .githooks/install.sh` (**not** `bash hooks/install.sh`, which HUMAN_GUIDE §12 states as the generic command). Git hooks live in `.git/hooks/` and never survive a clone, so this is a per-clone step.
