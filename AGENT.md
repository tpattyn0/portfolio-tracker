# AGENT.md — portfolio-tracker
> Project-specific conventions and constraints. Generic guardrails are in `CLAUDE.md` — not here.

## Conventions

- **Route handlers stay thin, delegate to `lib/services/*.service.ts`** (ADR-3). Don't put business logic directly in `app/api/**/route.ts`.
- **Auth guard pattern for new routes:** prefer `getAuthenticatedUser()` / `getAuthenticatedUserWithPortfolio()` from `lib/utils/auth.ts` over inlining `getServerSession(authOptions)` + a manual 401 — the shared helper is the single source of truth if the auth check ever changes. (An inline variant exists on 7 older routes; don't propagate it further.)
- **Ownership scoping is done correctly and consistently** — `getAuthenticatedUserWithPortfolio()` derives `portfolioId` from the session, position lookups are scoped by the compound `portfolioId_ticker` key, and wishlist mutations gate on a `findFirst` scoped to the user's own wishlist before writing (`app/api/portfolio/positions/[ticker]/sell/route.ts:28-39`, `lib/services/wishlist.service.ts:110-156`). Follow this pattern for any new mutation route; a reviewer should treat a route that skips it as a deviation worth flagging.
- **Rate limiting:** use `checkRateLimit(request, scope, maxAttempts, windowMs)` from `lib/middleware/rate-limit.ts` for unauthenticated or third-party-fanout routes. It's in-memory and per-instance (see fragile surfaces) — fine for now, revisit if deployed multi-instance.
- **Financial calculation services** (`intrinsic-value.service.ts`, `technical-analysis.service.ts`, `fundamental-analysis.service.ts`) are pure/static-method classes reading from cached DB tables or live Yahoo data. Keep them side-effect-free where possible; persistence/caching happens in the route or a thin wrapper, not buried inside the calculation itself.

## Known fragile surfaces

- **`lib/env.ts` and `lib/utils/env.ts` are both dead code** — neither is imported anywhere. Both validate `GOOGLE_API_KEY`, but the actual runtime env var is `GEMINI_API_KEY` (`lib/services/sentiment.service.ts:18-21`, `app/api/insights/portfolio/route.ts:46`). Do not wire either up without fixing the variable name first — doing so naively would validate the wrong key and could pass validation while Gemini silently fails.
- **`lib/services/sentiment.service.ts:17-22,190`** — the service constructor throws if `GEMINI_API_KEY` is unset, and a singleton instance is created at module scope. Importing this module anywhere without the env var set crashes at import time, not call time. Be careful adding new imports of this module in code paths that might run without the key configured (e.g. build-time static analysis, scripts).
- **`lib/services/wishlist.service.ts` previously made self-referential HTTP calls** to its own `/api/news/[symbol]` and `/api/research/[symbol]/intrinsic-value` routes. Fixed 2026-07-17 (AUD-05/TD-10): both now call `newsService.getAnalyzedNewsForSymbol()` and `IntrinsicValueService.calculateIntrinsicValue()` directly. The self-fetch had silently 401'd since those routes gained required auth (ONB-05, 2026-07-16) — `sentimentScore`/`intrinsicScore` were always `null` and the composite score always substituted a neutral 5 for 40% of its weight. If wishlist composite scores seem wrong, verify this path is still calling the services directly (not reverted to `fetch`).
- **`lib/services/fundamental-analysis.service.ts:78`** — cache freshness is gated partly by a hardcoded `latestMigrationDate` literal tied to a past scoring-logic change. Any future change to extraction/scoring logic needs this date bumped, or stale cached data will be served past the normal 24h TTL.
- **Two independent Gemini client instantiations** — `sentiment.service.ts` and `app/api/insights/portfolio/route.ts` each construct their own `GoogleGenerativeAI` client. Both now use `gemini-1.5-flash` as their only model (the insights route's nested-fallback try/catch to `gemini-1.5-pro`/`gemini-pro` was dead code — `getGenerativeModel` doesn't validate the model name, so it never threw there — removed 2026-07-17 per AUD-07). Keep this in mind if the model ever needs to change — it must change in both places, ideally by consolidating into one shared client (TD-12).
- **`IndustryComparison` table is read but never written** — `intrinsic-value.service.ts` reads it for peer-relative valuation but nothing populates it, so those methods fall back to hardcoded defaults (`industryPE = 15`, `industryPB = 1.5`). If this table is ever seeded, re-verify the fallback logic still behaves correctly.
- **`app/api/portfolio/closed-positions/route.ts`** — recomputes on every request (no caching). As of 2026-07-17 the FIFO matching and realized-P/L math live in `lib/services/realized-pl.service.ts`, shared with the sell route (fixed AUD-03: the two surfaces previously disagreed — avg-cost vs FIFO, inconsistent fee treatment; and AUD-04: `isPartial` was dead code because the loop `continue`d past unmatched sells instead of computing it). Both routes now have unit tests for the shared helper. Treat changes here carefully regardless; a bug is a silent wrong number shown to the user (win rate, realized P/L).
- **`app/api/portfolio/positions/[ticker]/sell/route.ts`** — matching a sell against BUY lots is not enough on its own (AUD-FIX-01, 2026-07-17): the route must also replay every **prior SELL** on the position through `matchFifoLots` first to deplete the lots before matching the new sell, or a second sell re-matches against already-consumed lots and `portfolio.realizedPL` silently disagrees with the closed-positions total. The closed-positions route gets this for free because it holds one `buyLots` array and depletes it in a single pass across all sells in chronological order — the sell route re-derives lot state from scratch on every call, so it must explicitly reconstruct "state as of just before this sell" by replaying prior sells. If you ever change how lots are loaded here, keep the prior-sell replay; `resolveCostBasisWithFallback` in `realized-pl.service.ts` is the shared unmatched-remainder fallback used by both routes — change the fallback logic there, not per-route.
- **`lib/services/position.service.ts`** (added 2026-07-17, AUD-02) — single source of truth for "buy more of an existing position": Decimal arithmetic, wrapped in the caller's `$transaction`, recalculates portfolio totals. Used by both `POST /api/portfolio/positions` (existing-position branch) and `POST /api/portfolio/positions/[ticker]/buy`. If either route's buy behavior needs to change, change it here — do not reintroduce a second implementation.
- **`lib/services/realized-pl.service.ts`** (added 2026-07-17, AUD-03; extended AUD-FIX-01/03) — single source of truth for FIFO lot matching, the unmatched-remainder average-cost fallback (`resolveCostBasisWithFallback`), and realized P/L (including buy and sell fees). Used by both the sell route and the closed-positions route. `matchFifoLots` mutates its `lots` array in place — pass a fresh array per call, and remember that "call it once per prior sell, then once for the new sell" is how the sell route reconstructs cumulative depletion (see the sell-route fragile-surface entry above).
- **`ComponentErrorBoundary` coverage is inconsistent** — only wraps sections on `dashboard/page.tsx` and `portfolio/[ticker]/page.tsx`. `research/[symbol]/page.tsx`, `research/page.tsx`, `wishlist/page.tsx`, `portfolio/add/page.tsx`, and `portfolio/closed-positions/page.tsx` have no boundary around components that depend on external API calls. A runtime error there crashes the whole page. `components/error-boundary.tsx` also exports a full-page `ErrorBoundary` class that is never used anywhere, including `app/layout.tsx`.
- **`research/[symbol]/page.tsx` doesn't use React Query for its main quote fetch** unlike every other data-fetching page — it fetches via a manual `useEffect` + `fetch`, missing caching/refetch-on-focus/shared invalidation that the rest of the app relies on. It does now (2026-07-18, MRD-Q1) use `useQuery` for a second, narrower purpose — the same `GET /api/portfolio/positions/[ticker]` ownership lookup `components/research/transactions-tab.tsx` already makes (404 = not held), reused to drive the Overview tab's `context` prop (`overviewContext`). Both call sites duplicate the same fetch/404 logic inline rather than sharing a hook — if a third consumer needs "is this symbol owned", extract a shared `usePositionQuery(symbol)` hook instead of a third copy.
- **Rotating any key in `.env`/`.env.local` can break `npm run verify`** — a new secret value may match a *different* gitleaks rule than the old one (the rotated Gemini key moved from `gcp-api-key` to `generic-api-key`, twice now: `.env` on 2026-07-17, `.env.local` the same day). The fix is a new narrow fingerprint in `.gitleaks-local/.gitleaksignore` for **each** env file carrying the key — never a path allowlist. Read that file's header before touching it.
- **`.next/` was tracked in git until 2026-07-16** — if you ever see hundreds of unrelated `.next/*` diffs in `git status`, someone re-added it; re-run `git rm -r --cached .next`. Do not `git add -A` in this repo.
- **The color token system (`app/globals.css` + `tailwind.config.js`) is load-bearing for all 20 `components/ui/*` files (ADR-8, 2026-07-17 Meridian overhaul).** Every shadcn CSS variable (`--background`, `--card`, `--border`, …) now holds a complete color value (hex or `oklch(...)`), **not** an HSL triple — `tailwind.config.js` maps them as `'var(--x)'`, with the `hsl()` wrapper removed. Do **not** re-add `hsl(var(--x))` to `tailwind.config.js`: it cannot parse hex or oklch, and re-adding it would silently break every component that consumes `bg-card`/`border-border`/etc. (all 20 files in `components/ui/`, Radix primitives included). If `npx shadcn add` ever generates a new component, its own inline `hsl(var(--x))` usage (if any) needs adjusting to match this convention — check the generated file against `components/ui/button.tsx` for the expected pattern (Tailwind alias, no inline `hsl()`). The Meridian-only tokens (`--mut`, `--line2`, `--fill`, `--up`, `--dn`, `--amber`, `--btnbg`, `--btnfg`) are also exposed as Tailwind color aliases (`text-mut`, `bg-fill`, etc.) — reference DESIGN.md's token table before adding a new one-off value.
- **The dashboard performance chart (`components/portfolio-chart.tsx`) is a hand-rolled inline SVG, not Recharts** (ADR-10) — it calls `buildPath`/`buildAreaPath` from `lib/utils/chart-path.ts`, a pure Catmull-Rom→bezier path builder with its own test file. If you change the chart's visual behavior, change the math in `chart-path.ts` (and its tests) — do not inline path-building logic back into the component.
- **The research-detail Overview/Technical charts (`components/research/detail-price-chart.tsx`) are a second, distinct inline-SVG chart** (ADR-11) — same `buildPath`/`buildAreaPath` primitives as the dashboard hero, but a separate component: no range-morph, and it adds a minimal y-axis (`niceYTicks` from `lib/utils/chart-ticks.ts`) + optional dashed reference lines that the hero chart intentionally does not have. `components/price-chart.tsx` (Recharts) was deleted 2026-07-18 once this component replaced its last importer — Recharts (`recharts` package) is no longer used anywhere in the app. **As of `plans/2026-07-18-meridian-dashboard-detail-fixes.md`, the dashboard hero chart (`components/portfolio-chart.tsx`) now also has hover crosshair+tooltip and y-axis price labels**, ported from this component's container-ref pixel-mapping approach and sharing `niceYTicks` — the two components are no longer distinguished by "has hover/y-axis vs. doesn't"; the hero still has its own gridline y-fractions (55/110/165 of a 220-tall viewBox, vs. this component's 47/94/141 of 190) and no range-morph interruption risk was introduced (the mousemove handler only reads `animatedValues`/`series` state, never touches the RAF ref). Do not merge the two SVG charts or backport the hero's range-morph into this one, or vice versa, without a Designer decision.
- **`components/research/*` shared primitives are the single source of truth for the research-detail tabs' editorial chrome** (ADR-11) — `HeadlineScoreCard`, `ScoreFigure`/`VerdictStamp`, `SubscoreBand`, `GradedMetricRow` are consumed by `overview.tsx`, `technical-analysis.tsx`, `fundamental-analysis.tsx`, `analyst-ratings.tsx`, `intrinsic-value.tsx`, `news-feed.tsx`. If a tab's headline card needs to look different, change the shared component (or add a prop) — do not fork a one-off variant inline in a tab component; that was the exact duplication `HeadlineScoreCard` was extracted to prevent.
- **Score/grade banding is centralized in `lib/utils/score-band.ts`** — `scoreBandClass`/`gradingDotClass`/`metricGrade` are presentational-only (recolor already-computed numbers against fixed thresholds) and must never be changed to alter scoring math; that lives in `lib/services/*-analysis.service.ts`. If a new tab needs a banded figure or grading dot, import these helpers rather than reimplementing the ≥7/4-7/<4 threshold inline.
- **`lib/services/technical-analysis.service.ts` has two entry points now** (`plans/2026-07-18-performance-audit-remediation.md` Task 3) — `calculateIndicators` (the pure, always-fresh calculation) and `getCachedIndicators(symbol, prices, volumes)` (a keyed 60s in-process memo wrapping it, matching `market-data.service`'s history cache TTL). `app/api/market/chart/[symbol]/route.ts` calls the cached wrapper; `wishlist.service.ts` still calls `calculateIndicators` directly (its own historical-data fetch is already 60s-cached in `market-data.service`, so a second cache layer there would key on a different fingerprint per call site for no benefit). If a new caller needs indicators for a stable `symbol`, prefer `getCachedIndicators` — but never change what either method computes; the cache is memoization only, not a second implementation.
- **`app/api/portfolio/route.ts`'s day-change now comes from `getQuote(...).previousClose`, not a per-position `getHistoricalRange` call** (Task 4, same plan) — if a future change needs the *prior official trading-day close* specifically (as distinct from Yahoo's `regularMarketPreviousClose`), that's a deliberate behavior change requiring a fresh parity check, not a silent revert of this fix.
- **`lib/services/wishlist.service.ts`'s five per-item score fetches run concurrently via `Promise.allSettled`, not sequential `await`s** (Task 5) — each dimension's failure is isolated (a rejected promise for one score does not affect the others), matching the pre-parallelization try/catch-per-dimension behavior. Do not reintroduce a sequential chain here; if a new score dimension is added, add it to the same `Promise.allSettled` array.
- **`app/(dashboard)/layout.tsx` must stay a non-blocking, synchronous Server Component — do not reintroduce a server-side session `await` there.** Page-route auth is enforced solely by `middleware.ts` (ADR-2, ADR-16); the layout previously duplicated that check via `await getServerSession(authOptions)` + `redirect`, which blocked the entire dashboard subtree — including the shared `<Navigation>` and every child route's `loading.tsx` Suspense fallback — from painting until the session check resolved. In the App Router, a segment's `loading.tsx` wraps only that segment's children in a Suspense boundary, not the layout above it, so any blocking `await` in this layout's render path re-freezes navigation across the whole group, not just the route being fetched. If a defence-in-depth check is ever wanted here again, it must be non-blocking (e.g. a separate Suspense-wrapped server component reading the session), never a render-gating `await` in the layout itself. See `plans/2026-07-19-meridian-nav-responsiveness.md` and ADR-16.
- **The dev server runs Turbopack (`next dev --turbopack`, ADR-17); `next.config.js` must stay free of a `webpack()` function.** Turbopack does not honor a custom webpack config function the way webpack does — adding one to fix a dev-only problem would silently not apply under `next dev`, while still applying to `next build` (which uses the default bundler, unaffected by this change). If a webpack-only customization is ever genuinely needed, it needs an explicit Turbopack (`turbopack`/`experimental.turbo`) equivalent in the same change, not a bare `webpack()` addition assumed to cover both.
- **`components/research/transactions-tab.tsx`'s live stat band must gate on `position.quantity > 0`, never on `!!position` alone** (ADR-18, `plans/2026-07-19-positions-tab.md`) — a fully-sold-but-not-deleted position still returns a 200 from `GET /api/portfolio/positions/[ticker]` with `quantity: 0` (the sell route updates quantity, it never deletes). Truthy-checking the position record alone would render a misleading zero-value Market value / Unrealised P/L panel for a closed position. The shared, tested helper `getPositionsPanelState(position)` in `lib/utils/positions-tab.ts` is the single source of truth for this three-way distinction (`"held"` / `"closed"` / `"none"`) — if a future caller needs "is this symbol currently held with live stats to show," use this helper, not an inline `!!positionQ.data` check. The companion helper `shouldShowPositionsTab(transactions)` is a separate, coarser gate (any transaction on file, regardless of quantity) driving whether the Positions tab appears at all — do not conflate the two: a symbol can have `shouldShowPositionsTab === true` and `getPositionsPanelState === "closed"` simultaneously (a fully-sold position with transaction history).
- **`lib/yahoo-finance.ts`'s `safeQuoteSummary` is the single sanctioned entry point for `yahooFinance.quoteSummary(...)`** (ADR-15, `plans/2026-07-18-yahoo-validation-error.md`) — it catches yahoo-finance2 v3's `FailedYahooValidationError` (thrown when Yahoo drifts its `quoteSummary` payload shape against the library's frozen Zod schema, a recurring failure mode) and falls back to the library's own coerced `error.result`, logging a `console.warn` drift signal. **Detection is by `error instanceof Error && error.name === "FailedYahooValidationError"`, never `instanceof yahooFinance.errors.FailedYahooValidationError`** — the installed `yahoo-finance2@3.13.0` does not expose an `errors` subpath in its package `exports` map, so that pattern is unavailable, and a deep import of the error class (`yahoo-finance2/esm/src/lib/errors.js`) would violate the exports map and can break Next.js bundling. All three `quoteSummary` call sites (`fundamental-analysis.service.ts`, `market-data.service.ts`, `analyst-ratings.service.ts`) route through this wrapper — any new call site must too, or it silently regresses to the pre-fix 500-on-drift behavior. `market-data`/`fundamentals` each throw if their structurally-required module (`price`/`summaryDetail`) is absent from the wrapper's return (fail loud rather than persist/cache an all-null row); `analyst-ratings` deliberately does not add that guard, since a missing `recommendationTrend` is a valid "no analyst coverage" state already handled by the existing neutral-score path. The `console.warn` drift line is the signal to eventually bump the `yahoo-finance2` dependency.

## Hard limits

- Dev and production share one database (ADR-6) — be extra cautious with any destructive script, seed, or migration; there is no isolated dev DB to test against yet.
- Do not wire up `lib/middleware/csrf.ts`-style CSRF protection without also building a token-issuance flow — a prior version of this file was deleted as dead code because the cookie it expected was never set anywhere (see `TECH_DEBT.md`).

## Verify

Single command, run before every commit:

```bash
npm run verify
```

This runs, in order: `typecheck` (`tsc --noEmit`), `lint` (`next lint`), `test` (`vitest run`), and `secret-scan` (`gitleaks detect --source . --no-git -c .gitleaks.toml --redact`). All four must pass. `lint` currently has pre-existing warnings (unused imports, a few `any` types) that do not fail the command — do not treat fixing all of them as in-scope for an unrelated change; fix only what you touch.

### Reading CI on a PR

Two checks, and only one of them blocks:

| Check | Blocks merge? | What a red means |
|---|---|---|
| `verify / code-gate` | **Yes — required** | A real failure. Typecheck, lint, tests, or a secret in *this PR's* commits. |
| `verify / secret-history (reporting only — expected red)` | No | **Expected.** It reports the known historical leak (ADR-7 / TD-01). It is red on every run and will stay red until TD-01 is fully resolved. |

So a PR showing one red X and one green is **normal**. Two reds means `code-gate` failed and something is actually wrong. These were previously both named `verify`, which made a real failure indistinguishable from the expected one at a glance — renamed 2026-07-17.

**CI runs `npm run verify:code`, not `npm run verify`** — same typecheck + lint + test gate, minus the local secret scan. CI handles secrets in its own steps instead: `code-gate` scans this PR's commit range, and `secret-history` scans all of history. Neither uses the gitleaks GitHub Action — on a `pull_request` event it only scans the PR's own commits and ignores `fetch-depth`, which is how a green check once coexisted with three live secrets in the first commit. Keep the two in sync: a check added to `verify` belongs in `verify:code` too.

## Git hooks live in `.githooks/`, not `hooks/`

`hooks/` is application source (React hooks — `use-debounce`, `use-price-sync`, `use-toast`), imported as `@/hooks/*` across `app/` and `components/`. The workflow framework's rollout script defaults to putting `pre-commit` and `install.sh` in a root `hooks/` directory; in this repo they were moved to `.githooks/` to keep shell scripts out of a source tree.

Install them on a fresh clone with `bash .githooks/install.sh` (**not** `bash hooks/install.sh`, which HUMAN_GUIDE §12 states as the generic command). Git hooks live in `.git/hooks/` and never survive a clone, so this is a per-clone step.
